# Sendtrap Community — parity coverage evidence

Every row of the approved feature-parity matrix mapped to the Community mechanism and the evidence
that the mechanism works **under Community's own bindings**. "Package suite" = the test suite carried
by `sendtrap/core` itself (row 33); "Community suite" = this repo's `tests/Feature/*`.

Dispositions: **Parity** (behavior-identical), **Adapted** (documented difference),
**Cloud-only** (excluded by construction), **Dev-only** (ships nowhere).

## Known gaps / honest notes for the gate review

- **Row 18**: `laravel/reverb` is not installed in Phase 4b; live updates are a documented config-flip
  (`.env.example`) on top of the tested lazy-Echo degrade path. Running-server wiring is Phase 7/8
  container work per §6.6/§12.2.
- **Rows 14/15/17**: no Community-side behavioral test by design — the mechanisms carry zero Community
  code; coverage lives in the package suite (row 33). Community proves the *bindings* those jobs consume
  (rows 5/6/13).
- **Row 21**: the email-verification challenge flow is enabled but unreachable in practice (all users are
  provisioned verified); no test drives the verification email round trip.
- **Row 1**: `config/sendtrap.php` is *merged* from the package rather than published into `config/` —
  every key resolves (verified); publish with `--tag=sendtrap-config` to override locally.
- **D-13 / D-17**: matrix approval and limit-zero semantics remain open owner decisions; D-17's
  provisional convention (absent = unlimited, `0` = blocked) is pinned in both directions by tests and a
  reversal is a comment + expected-direction flip (design §5.2).
- **Phase 6 (not this slice)**: LICENSE / CONTRIBUTING / README product docs, repo publication, tagged
  package dependency (currently a path symlink to the live worktree — design §12.3).
